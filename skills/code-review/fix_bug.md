# Fix Bug from PR Review Comments

Analyze all review comments on a PR, judge their validity, apply code fixes, reply to each comment, and produce a structured fix report.

---

## Trigger

Use this skill when the user says:

- "fix review comments"
- "fix bug from review"
- "handle PR review comments"
- "address review feedback"
- "reply to review comments"
- "fix issues from review.json"
- "process local review results"
- "帮我修复 review 评论" / "处理 PR 上的 review 意见" / "根据 review 修 bug" / "回复 review 评论" / "处理本地 review 结果"

---

## Two Modes

| Mode | Source | When to use |
|------|--------|-------------|
| **Local** | `review.json` produced by `lazyllm review-local` | Fixing your own code before pushing |
| **PR (online)** | Review comments on a remote PR | Addressing feedback from others on GitHub/GitLab/Gitee |

---

## Mode A: Local Review (review.json)

Use this mode when you have a `review.json` file generated by `lazyllm review-local`.

### Step A-1: Generate review.json (if not already done)

```bash
# Review uncommitted + committed changes against main
lazyllm review-local --base main --output review.json

# Committed changes only
lazyllm review-local --base main --no-uncommitted --output review.json

# Force restart (clear checkpoint)
lazyllm review-local --base main --clear-checkpoint --output review.json
```

### Step A-2: Read and understand review.json

```json
{
  "instructions": "...(AI processing guide — follow directly)",
  "pr_info": { "branch": "feature/xxx", "base": "main", "summary": "..." },
  "stats": { "total": 5, "critical": 1, "normal": 3, "suggestion": 1 },
  "issues": [
    {
      "id": 1, "file": "src/foo.py", "line": 42,
      "severity": "critical", "category": "correctness",
      "problem": "Description of the problem",
      "suggestion": "Suggested fix"
    }
  ]
}
```

- Follow the embedded `instructions` field directly
- Process by priority: `critical` → `normal` → `suggestion`

### Step A-3: Judge and fix each issue

| Situation | Verdict | Action |
|-----------|---------|--------|
| Real bug or security issue | Reliable | Fix the code |
| Already fixed elsewhere | Not Reliable | Record, skip |
| AI misunderstood the logic | Not Reliable | Record rejection reason |
| Pre-existing issue (not from this change) | Case-by-case | Fix if trivial, open issue if complex |
| Style conflict with project conventions | Not Reliable | Record, skip |

Use `file` + `line` to locate the code, apply the fix guided by `suggestion`. Only change code directly related to the issue — preserve existing style.

### Step A-4: Generate fix report

```markdown
# Review Fix Report (Local)

## Overview
- Issues scanned: N (critical: X, normal: Y, suggestion: Z)
- Fixed: A | Rejected: B | Skipped: C (suggestion, optional)

## Fixed

### [critical] Issue #1 — src/foo.py:42
**Problem**: Null dereference — crashes when result is None
**Fix**: Added None guard, returns default value

## Rejected (Not Reliable)

### Issue #2
**Reason**: False positive — variable is guaranteed non-null by the caller

## Unresolved (suggestion level, optional)
- Issue #5: Consider adding type annotations (low priority)
```

## Mode B: PR Review (online)

Use this mode when fixing review comments on a remote PR (GitHub / GitLab / Gitee).

### Prerequisites

- `gh` CLI installed and authenticated (`gh auth login`)
- Read/write access to the target PR
- PR number and repository name (`owner/repo` format)

---

## Step B-1: Fetch All Review Comments

**Always use `gh` CLI — never browse the PR page directly.** Browsing may miss collapsed or outdated comments.

```bash
# Line-level review comments
gh api repos/{OWNER}/{REPO}/pulls/{PR}/comments

# Overall reviews (non-line-level)
gh api repos/{OWNER}/{REPO}/pulls/{PR}/reviews

# Comments under a specific review
gh api repos/{OWNER}/{REPO}/pulls/{PR}/reviews/{review_id}/comments
```

> Merge all three sources to ensure full coverage. Do NOT skip any comment.

---

## Step B-2: Judge Each Comment

For every comment, determine whether it is **Reliable** or **Not Reliable**:

| Verdict | Meaning |
|---------|---------|
| Reliable | The comment identifies a real issue — fix it |
| Not Reliable | The comment is based on a misunderstanding, incorrect assumption, outdated context, already-fixed issue, or inapplicable constraint |

**Common Not Reliable cases:**

- Reviewer misunderstood the code logic
- Issue was already fixed in a later commit
- Comment is based on stale context
- Constraint does not apply to this project
- Pure style preference with no functional impact

**Key rules:**

- Do not accept a comment just because "others said so"
- Architectural issues introduced by this PR **must be fixed**
- Pre-existing issues: fix if trivial, otherwise open/link a tracking issue
- Always verify whether the issue is already resolved before acting
- Avoid over-fixing: only modify code when the comment is valid and actionable

---

## Step B-3: Apply Code Fixes

For each Reliable comment, apply a fix at the indicated `file` + `line`.

- Only change what the comment targets — avoid unrelated modifications
- Verify correctness after fixing
- If the fix has broad impact, update related tests accordingly

---

## Step B-4: Reply to Each Comment

After fixing, reply to every comment via `gh` — both fixed and rejected ones.

```bash
# Reply to a line-level review comment
gh api --method POST \
  repos/{OWNER}/{REPO}/pulls/{PR}/comments \
  --field in_reply_to={comment_id} \
  --field body="Fixed: {brief description of the fix}"

# Reply to an overall review (non-line-level)
gh api --method POST \
  repos/{OWNER}/{REPO}/issues/{PR}/comments \
  --field body="All review comments have been addressed. See fix report below."
```

For Not Reliable comments, reply with a clear rejection reason.

---

## Step B-5: Generate Fix Report

After processing all comments, output a structured report:

```markdown
# Review Fix Report (PR)

## Summary
- Total comments: N | Fixed: X | Rejected: Y | Not Applicable: Z

## Fixed

### [Category: Correctness / Performance / Architecture / Style / Maintainability / ...]

| Comment ID | File | Line | Fix Description |
|------------|------|------|-----------------|
| 1 | foo.py | 42 | Added None guard to prevent null dereference |

## Rejected (Not Reliable)

| Comment ID | Reason |
|------------|--------|
| 3 | Already fixed in commit abc123 |

## Unresolved

(If any — explain why, e.g. needs more context, tracked in issue #xx)
```

**Report requirements:**

- Every comment must appear in the report — no omissions
- Group by category (correctness, performance, architecture, style, maintainability, etc.)
- List each comment individually — do not merge
- Maintain traceability: every comment must be independently locatable and verifiable

## Full Examples

### Local review

```bash
# 1. Run review-local to generate review.json
lazyllm review-local --base main --output review.json

# 2. AI reads review.json, judges and fixes issues

# 3. Commit fixes
git add -A && git commit -m "fix: address local review issues"
git push
```

### PR review (online)

```bash
# 1. Fetch comments
gh api repos/LazyAGI/LazyLLM/pulls/42/comments > comments.json

# 2. AI analyzes and applies fixes

# 3. Reply to a fixed comment
gh api --method POST \
  repos/LazyAGI/LazyLLM/pulls/42/comments \
  --field in_reply_to=123456 \
  --field body="Fixed: changed type of \`x\` from \`str\` to \`Optional[str]\` and added a None check."

# 4. Commit and push
git add -A && git commit -m "fix: address review comments on PR #42"
git push
```
